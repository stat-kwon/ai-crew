# 요구사항 문서: ai-crew 하네스 엔지니어링 코어 강화

## Intent Analysis

- **사용자 요청**: 하네스 엔지니어링을 위해 실제 코드 기반으로 부족한 부분을 파악하고 개선. 참고 프로젝트(ECC, OMC)와 비교 분석 후 도입할 기능을 설계
- **요청 유형**: Enhancement (기존 시스템 개선)
- **범위**: System-wide (코어 하네스 전반)
- **복잡도**: Complex (다수 모듈에 걸친 변경, 새 기능 추가)
- **이터레이션 범위**: Phase 1 — 코어 강화 (안정성 중심)

---

## 기능 요구사항 (Functional Requirements)

### FR-1: Install-State 추적
**우선순위**: 높음

현재 ai-crew는 번들 설치 후 무엇이 설치되었는지 추적하지 않음. install-state를 기록하여 진단/제거 기능을 제공해야 함.

- **FR-1.1**: 설치 시 `.ai-crew/install-state.json`(프로젝트별)에 설치 내역 기록 (번들명, 설치된 파일 목록, 타임스탬프, 버전)
- **FR-1.2**: `ai-crew doctor` 커맨드 — 설치 상태 진단 (누락 파일, 설정 불일치 감지)
- **FR-1.3**: `ai-crew uninstall` 커맨드 — install-state 기반으로 설치된 파일 정리 제거

### FR-2: 에이전트 선언 모델 기본값
**우선순위**: 중간

현재 에이전트 정의(.md)에 모델 선호도 필드가 없음. 에이전트가 적합한 모델 티어를 선언할 수 있어야 함.

- **FR-2.1**: 에이전트 정의 마크다운에 YAML 프론트매터 `model` 필드 추가 (예: `model: opus`, `model: sonnet`)
- **FR-2.2**: graph.yaml의 노드별 `config.model`이 에이전트 기본값을 오버라이드
- **FR-2.3**: 우선순위: graph.yaml > 에이전트 선언 > bundle defaults

### FR-3: 경량 검증 프로토콜
**우선순위**: 높음

현재 노드 완료 시 검증 없이 "completed" 처리됨. 노드별 선택적 검증을 지원해야 함.

- **FR-3.1**: graph.yaml 노드에 `verify` 필드 추가 (예: `verify: [test, lint, build]`)
- **FR-3.2**: 검증 타입별 실행 로직: `test` → npm test, `lint` → npm run lint, `build` → npm run build (설정 가능)
- **FR-3.3**: 검증 실패 시 노드 상태를 `failed`로 전환, 에러 메시지를 스크래치패드에 기록
- **FR-3.4**: `verify` 필드가 없는 노드는 기존 동작 유지 (검증 스킵)

### FR-4: 에이전트 간 공유 메모리
**우선순위**: 중간

현재 에이전트 간 상태 전달은 스크래치패드 파일(.md)로만 가능. 구조화된 공유 메모리를 제공해야 함.

- **FR-4.1**: `.ai-crew/shared-memory.json`에 노드별 구조화된 출력 저장
- **FR-4.2**: 쓰기: 에이전트가 노드 완료 시 key-value 형태로 결과 기록
- **FR-4.3**: 읽기: 하위 노드가 실행 시 의존 노드의 공유 메모리를 자동 로드
- **FR-4.4**: 스크래치패드(.md)는 사람 리뷰용으로 유지, 공유 메모리(.json)는 에이전트 간 기계 읽기용

### FR-5: 기본 재시도 및 자동 상태 저장
**우선순위**: 높음

현재 실패 노드에 재시도 없이 그래프 실행이 중단됨. 기본적인 내결함성을 제공해야 함.

- **FR-5.1**: graph.yaml 노드에 `retry` 필드 추가 (기본값: 0, 최대: 3)
- **FR-5.2**: 재시도 시 이전 워크트리 정리 후 새로 시작
- **FR-5.3**: 노드 완료 시 자동으로 state.json 업데이트 (현재도 하지만 즉시 디스크에 flush 보장)
- **FR-5.4**: 각 레벨 완료 시 자동 체크포인트 생성 (`.ai-crew/checkpoints/level-{N}.json`)

### FR-6: 훅 프로파일링
**우선순위**: 중간

현재 설치된 훅이 무조건 전부 실행됨. 번들/상황별로 훅을 선택적으로 활성화할 수 있어야 함.

- **FR-6.1**: `config.yaml`에 `hookProfile` 필드 추가 (예: `minimal`, `standard`, `strict`)
- **FR-6.2**: 각 훅 정의에 `profiles` 필드 추가 — 어떤 프로필에서 활성화되는지 선언
- **FR-6.3**: 환경변수 `AI_CREW_HOOK_PROFILE`로 런타임 오버라이드
- **FR-6.4**: 훅에 `timeout` 필드 추가 (기본: 30초, 최대: 120초)

### FR-7: 스키마 검증
**우선순위**: 중간

현재 bundle.yaml, graph.yaml 등의 선언적 설정에 형식 검증이 없음. 잘못된 설정을 조기에 감지해야 함.

- **FR-7.1**: JSON Schema 정의: `bundle.yaml`, `graph.yaml`, `plugin.json`, `hooks.json`, `state.json`
- **FR-7.2**: `ai-crew install` 시 번들 스키마 자동 검증
- **FR-7.3**: `ai-crew validate` 커맨드 — 설치된 설정의 스키마 검증
- **FR-7.4**: 검증 실패 시 명확한 에러 메시지와 수정 가이드

### FR-8: 경량 세션 학습
**우선순위**: 낮음

같은 프로젝트에서 반복 실행 시 에이전트가 매번 동일한 코드베이스를 처음부터 파악함. 프로젝트 컨텍스트를 캐싱하여 재활용해야 함.

- **FR-8.1**: 그래프 실행 완료 후 프로젝트 컨텍스트 요약을 `.ai-crew/project-context.json`에 저장
- **FR-8.2**: 저장 내용: 기술 스택, 주요 패턴, 에이전트별 메모, 이전 실행에서의 학습
- **FR-8.3**: 다음 그래프 실행 시 에이전트 프롬프트에 프로젝트 컨텍스트 자동 주입
- **FR-8.4**: 컨텍스트는 수동으로 편집/삭제 가능

### FR-9: 라우터 노드 인터페이스 정의
**우선순위**: 낮음

현재 타입 시스템에 `router` 노드 타입이 있지만 구현이 없음. 인터페이스를 명확하게 정의해야 함.

- **FR-9.1**: `GraphNode` 타입의 `condition` 필드 형식 명세 (예: `"{node}.status == 'completed'"`)
- **FR-9.2**: graph.yaml에서 라우터 노드 선언 시 스키마 검증
- **FR-9.3**: 실제 조건 평가 로직은 이번 스코프에서 미구현 (stub만 제공, 경고 로그 출력)

## 비기능 요구사항 (Non-Functional Requirements)

### NFR-1: 하위 호환성
- 기존 bundle.yaml, graph.yaml 형식과 100% 하위 호환
- 새 필드(`verify`, `retry`, `model` 등)는 모두 선택적(optional)
- 기존 설치된 프로젝트가 업그레이드 후에도 정상 동작

### NFR-2: 테스트 커버리지
- 모든 새 기능에 대해 vitest 단위 테스트 작성
- 기존 테스트가 깨지지 않아야 함

### NFR-3: 코드 품질
- 기존 코드 스타일 준수 (ESM, TypeScript strict)
- 새 모듈은 단일 책임 원칙 준수
- 외부 의존성 최소화

### NFR-4: 성능
- 스키마 검증은 설치/검증 시점에만 실행 (런타임 영향 없음)
- 공유 메모리 JSON은 경량 유지 (노드당 최대 10KB 권장)
- 체크포인트 생성은 비동기 가능

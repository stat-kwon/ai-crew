## Question 1
기존 7개 페이지 중 4개(팀 편집, 환경 점검, 팀 템플릿, 설정)를 제거하는 것에 동의하시나요?

A) 전부 제거 — 제안서대로 3페이지 구조로 진행
B) 설정만 유지 — 설정 페이지는 별도로 남기고 나머지 3개 제거
C) 모두 유지하되 축소 — 히스토리가 메인이고 나머지는 서브메뉴로

## Question 2
사이드바를 제거하고 상단 탭 바로 전환하는 것에 대해 어떻게 생각하시나요?

A) 상단 탭 바로 전환 — 제안서대로 (콘텐츠 영역 최대화)
B) 사이드바 유지하되 축소 — 아이콘만 보이는 미니 사이드바 (3개 메뉴)
C) Other (please describe after [Answer]: tag below)

[Answer]: A
> 근거: 메뉴가 2개 탭 + 설정 아이콘뿐이므로 사이드바를 유지할 이유가 없음. 콘텐츠 영역 256px 확보.

## Question 3
타임라인(홈) 페이지에서 "현재 진행 중인 런"은 노드 미니맵을 확장 표시하고, 과거 런은 접힌 한 줄 요약으로 보여주는 방식에 동의하시나요?

A) 동의 — 진행 중 런만 확장, 나머지 접힘
B) 최근 3개까지 확장 — 진행 중 + 최근 완료 2개도 확장 표시
C) 모두 접힘 — 클릭 시에만 확장 (일관된 UX)
D) Other (please describe after [Answer]: tag below)

[Answer]: A
> 근거: 히스토리 관리 목적상 과거 런은 빠른 스캔이 중요. 진행 중 런만 실시간 상태가 필요하므로 확장. 과거는 필요 시 클릭으로 드릴다운.

## Question 4
런 상세 페이지에서 노드 결과물(scratchpad)을 어떤 방식으로 보여주길 원하시나요?

A) 인라인 아코디언 — 노드 카드를 클릭하면 바로 아래에 펼침 (제안서대로)
B) 우측 패널 — 노드 클릭 시 우측 문서 패널 영역에 표시
C) 전체 화면 모달 — 노드 클릭 시 풀스크린 마크다운 뷰어
D) Other (please describe after [Answer]: tag below)

[Answer]: A
> 근거: 아코디언은 컨텍스트 전환 없이 스크롤로 여러 노드를 비교할 수 있어 히스토리 탐색에 최적. 모달은 전체 맥락을 잃게 만듦.

## Question 5
설계 문서 페이지는 기존 구조(좌측 트리 + 우측 뷰어)를 유지하면서 스타일만 조정하는 것에 동의하시나요?

A) 기존 구조 유지 + 스타일 조정 — 제안서대로
B) 구조도 변경 — 폴더 트리 대신 카드 그리드로
C) Other (please describe after [Answer]: tag below)

[Answer]: A
> 근거: 좌측 트리 + 우측 뷰어는 문서 탐색의 표준 패턴. 이미 잘 동작하는 구조를 굳이 바꿀 필요 없음. 상단 탭 바 연동 + 스타일 통일만 필요.

## Question 6
런 상세에서 "이 런에서 생성된 문서" 패널을 우측에 표시하는 방식은 현재 API로 런↔문서 매핑이 완벽하지 않을 수 있습니다. 어떤 접근이 좋을까요?

A) 런 시점 기준 aidlc-docs 전체 문서를 표시 — 완벽한 매핑 없이도 가능
B) manifest.json에 문서 목록 필드를 추가 — 정확한 매핑 (백엔드 수정 필요)
C) 문서 패널 생략 — 런 상세는 노드 결과물만 표시, 문서는 설계 문서 탭에서
D) Other (please describe after [Answer]: tag below)

[Answer]: C
> 근거: 런↔문서 매핑이 불완전한 상태에서 부정확한 정보를 표시하면 혼란. 런 상세는 "에이전트 결과물" 확인이라는 명확한 역할에 집중. 문서는 설계 문서 탭이라는 전용 공간이 이미 있음. 향후 manifest 연동이 준비되면 추가 가능.

## Question 7
비주얼 톤은 현재 스타일(화이트 배경 + 인디고 주색 + shadcn/ui)을 유지하면서 레이아웃만 변경하는 것에 동의하시나요?

A) 현재 비주얼 톤 유지 — 색상, 폰트, 카드 스타일 그대로
B) 다크 모드 추가 — 현재 스타일 + 다크 모드 토글
C) 전면 비주얼 변경 — Linear/Vercel 스타일로 전환
D) Other (please describe after [Answer]: tag below)

[Answer]: A
> 근거: 이번 리디자인의 핵심은 정보 아키텍처(IA) 변경이지 비주얼 리프레시가 아님. 레이아웃 전면 변경 + 비주얼 변경을 동시에 하면 scope가 폭발. 단계적 접근: 1차 IA 변경 → 2차 비주얼 개선(필요시).

## Question 8: Security Extensions
Security extension 규칙을 이 프로젝트에 적용하시겠습니까?

A) Yes — 모든 SECURITY 규칙을 blocking 제약 조건으로 적용 (프로덕션급 애플리케이션 권장)
B) No — SECURITY 규칙 건너뜀 (PoC, 프로토타입, 실험적 프로젝트에 적합)
C) Other (please describe after [Answer]: tag below)

[Answer]: B
> 근거: 읽기 전용 로컬 대시보드. 외부 네트워크 노출 없음, 인증 불필요, 민감 데이터 없음. 이전 사이클에서도 B 선택.
